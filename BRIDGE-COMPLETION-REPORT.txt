# PROJECT HANDOFF — OPEN RIGHT NOW? (NANO APP)
## Mission
Design and plan a nano web app called **“Open Right Now?”**

This app allows a small business to instantly publish its current status to customers.

The purpose is extremely simple:

> Let customers know if you are open, closed, or temporarily unavailable — and why.

This is a **nano-app**, not a micro-app.
It must be:
- extremely small
- extremely fast
- extremely clear
- extremely easy to use

This project is part of an AI bridge test.
The planning output must be structured, explicit, and directly usable by a builder AI.

The final implementation target is a **mobile-first web app**.

---

## Core Product Vision

This app replaces:
- “Check Instagram to see if they’re open”
- “Call and hope they answer”
- “Drive there and find out they’re closed”

With:

> One simple page that always answers the question immediately.

---

## Product Behavior (Extremely Important)

There is only **ONE active state at a time**.

This is not a list app.
This is not a feed.
This is not a dashboard.

This is a **single live status**.

---

## Required Status Options

The app must support the following states:

- Open
- Closed
- Opening Late
- Back Soon
- Sold Out

The planning must define:
- how these are stored
- how they are displayed
- how they map to visual styles (color or icon)

---

## Required Data Model

The system must revolve around a **single current status record**.

Fields must include:

- business_name
- status
- note (optional)
- last_updated_timestamp

Optional:
- last_updated_by (if useful but not required)

---

## Required MVP Capabilities

### 1. Owner Update View
A user can:
- select a status (tap-based, not dropdown if possible)
- enter a short optional note
- save/update instantly

The update process must take under 10 seconds.

---

### 2. Public View
A user visiting the page sees:

- business name
- current status (very prominent)
- optional note
- “last updated X minutes ago”

It must be readable in under 2 seconds.

---

### 3. Persistence
The latest status must persist across:
- refresh
- server restart

---

### 4. Mobile-First UX
This app must feel effortless on a phone:
- big buttons
- minimal typing
- strong contrast
- no clutter

---

## Explicit Non-Goals
Do NOT include:

- multiple businesses
- user accounts (unless extremely lightweight)
- scheduling system
- analytics
- notifications
- chat
- history logs (optional but not required)
- anything that turns this into a “system”

This is a **single-state publisher**.

## Information Architecture

The app should have only two primary views:

1. Public View (default)
2. Owner Edit View

Optional:
- a hidden route or toggle to switch between them

The app should feel like:

- a sign on a door
- but digital
- and always up to date

Status should be:
- large
- obvious
- color-coded

- Green = Open

- Orange = Sold Out

## Edge Cases

Plan for:

- no status set yet
- empty note
- long note input
- rapid updates
- refresh mid-update
- weird characters in note
- user never updating again (stale timestamp)

# Project Blueprint for "Open Right Now?" Nano App

## Overview
The **"Open Right Now?"** nano app is a simple web application designed to provide real-time status updates for small businesses. The main goal is to allow businesses to communicate their current operational status to customers in a clear and efficient manner. The app will support five statuses: Open, Closed, Opening Late, Back Soon, and Sold Out. It will feature a mobile-first design to ensure ease of use on smartphones.

## Technical Architecture
The app will be built using the following technology stack:
- **Frontend**: React with Vite for fast development and a responsive user interface.
- **Backend**: Node.js with Express for handling API requests.
- **Database**: SQLite for lightweight data persistence.

### Architecture Diagram
```
+----------------+       +----------------+       +----------------+
|   User Device   | <--> |     Frontend   | <--> |     Backend    |
| (Browser/Mobile)|       | (React + Vite) |       | (Node + Express)|
+----------------+       +----------------+       +----------------+
                                 |
                                 |
                             +----------------+
                             |    SQLite DB   |
                             +----------------+
```

## MVP Scope
The Minimum Viable Product (MVP) will include the following features:
1. **Owner Update View**: A simple interface for business owners to select their current status and enter an optional note.
2. **Public View**: A display for customers showing the business name, current status, optional note, and last updated timestamp.
3. **Persistence**: The latest status must persist across page refreshes and server restarts.
4. **Mobile-First UX**: The app must be optimized for mobile devices with large buttons and a clean layout.

### Data Model
The data model for the application will consist of a single record with the following schema:
```sql
CREATE TABLE status_record (
  id INTEGER PRIMARY KEY DEFAULT 1, -- Singleton pattern
  business_name TEXT DEFAULT 'My Business',
  status TEXT CHECK(status IN ('Open', 'Closed', 'Opening Late', 'Back Soon', 'Sold Out', NULL)),
  note TEXT NULL,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
- **id**: Fixed as 1 for singleton.
- **business_name**: Default value is "My Business".
- **status**: ENUM with possible values as specified.
- **note**: Optional text field for additional information.
- **updated_at**: Timestamp of the last update.

## Implementation Phases
The implementation will be divided into the following phases:
### Phase 1: Setup Project Structure
- Initialize the project with Vite for the frontend.
- Set up Node.js and Express for the backend.
- Configure SQLite for data persistence.

### Phase 2: Build Frontend
- Create the Owner Update View with buttons for status selection and a text input for notes.
- Build the Public View to display the current status and last updated timestamp.
